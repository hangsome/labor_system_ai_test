## 2. 系统定位

- 项目名称：劳务系统管理后台
- 系统类型：中大型企业管理平台（Web 管理端）
- 目标用户：劳务公司管理人员、用工单位管理人员、财务与运营管理人员
- 一句话目标：把“线索-合同-考勤-结算-开票-资金”全链路数字化并可审计。

## 3. 范围定义（V1）

1. 统一认证与 RBAC（含数据权限）
2. 客户线索、用工单位、项目与合同管理
3. 员工全生命周期管理（入离职、档案、账户、保险）
4. 排班考勤、补卡审批、台账生成
5. 结算、支付、发票、应收与核销
6. 培训、绩效、积分与审计能力

## 4. 用户故事与验收标准（GIVEN-WHEN-THEN）

### 4.1 认证与权限

**US-AUTH-001**  
作为系统管理员，我希望按角色配置菜单与接口权限，以便限制不同岗位访问范围。  
验收标准：
- GIVEN 管理员已登录且有权限管理角色  
  WHEN 新建“财务专员”角色并分配权限  
  THEN 该角色用户仅可访问财务菜单与对应 API。

**US-AUTH-002**  
作为部门负责人，我希望数据按组织范围可见，以便避免越权查看。  
验收标准：
- GIVEN 用户的数据权限为 `DEPT`  
  WHEN 查询客户线索列表  
  THEN 仅返回其所属部门的数据。

### 4.2 客户与合同

**US-CRM-001**  
作为业务开发，我希望录入客户线索并维护跟进记录，以便跟踪转化。  
验收标准：
- GIVEN 我在客户线索页面填写必填项  
  WHEN 点击保存  
  THEN 系统创建线索并记录创建人、创建时间与初始状态。

**US-CRM-002**  
作为合同管理员，我希望创建 A/B 类合同并配置结算规则，以便自动生成后续结算计划。  
验收标准：
- GIVEN 合同基础字段和结算规则合法  
  WHEN 提交合同  
  THEN 系统成功保存并生成对应的收款计划。

### 4.3 员工与组织

**US-EMP-001**  
作为人事专员，我希望管理员工入职信息与银行账户，以便支持发薪。  
验收标准：
- GIVEN 员工基础信息和身份证校验通过  
  WHEN 完成入职提交流程  
  THEN 员工状态变为“在职”，并可绑定银行账户。

**US-EMP-002**  
作为人事专员，我希望记录员工离职并保留历史轨迹，以便审计。  
验收标准：
- GIVEN 员工状态为在职  
  WHEN 发起离职流程并审批通过  
  THEN 员工状态更新为离职，历史记录可追溯且不可篡改。

### 4.4 考勤与补卡

**US-ATT-001**  
作为排班管理员，我希望配置班次并导入排班，以便进行考勤核算。  
验收标准：
- GIVEN 班次模板已配置  
  WHEN 批量导入排班  
  THEN 系统校验并落库，生成排班记录。

**US-ATT-002**  
作为员工，我希望提交补卡申请并被主管审批，以便纠正异常考勤。  
验收标准：
- GIVEN 员工存在异常打卡  
  WHEN 员工提交补卡申请  
  THEN 申请进入审批流并在通过后触发台账重算。

### 4.5 结算与支付

**US-SETTLE-001**  
作为财务专员，我希望按合同规则自动生成结算单，以便减少人工计算。  
验收标准：
- GIVEN 结算周期到达且考勤数据完整  
  WHEN 触发结算任务  
  THEN 生成结算单与明细，并记录版本。

**US-SETTLE-002**  
作为出纳，我希望支付请求幂等，避免重复扣款。  
验收标准：
- GIVEN 相同 `X-Idempotency-Key` 的支付请求重复提交  
  WHEN 系统处理请求  
  THEN 只产生一次有效支付记录。

### 4.6 发票与应收

**US-AR-001**  
作为财务专员，我希望发票申请审批后自动进入应收台账，以便统一跟踪回款。  
验收标准：
- GIVEN 发票申请状态为 `APPROVED`  
  WHEN 财务开票  
  THEN 系统写入发票记录并创建应收台账条目。

**US-AR-002**  
作为财务专员，我希望支持分次核销并防止超额核销，以便保持账务准确。  
验收标准：
- GIVEN 现有未核销应收金额 > 0  
  WHEN 提交核销记录  
  THEN 系统校验累计核销金额不超过应收金额。

## 5. 业务规则（IF-THEN）

| 规则 ID | IF | THEN |
|---|---|---|
| BR-001 | 用户未通过认证 | 拒绝访问并返回 401 |
| BR-002 | 用户角色无接口权限 | 返回 403 |
| BR-003 | 数据权限为 `SELF` | 仅可访问本人创建或归属数据 |
| BR-004 | 合同状态为 `LOCKED` | 禁止直接修改核心结算字段 |
| BR-005 | 合同到期前 60 天 | 触发续签提醒任务 |
| BR-006 | 补卡申请审批通过且未封板 | 触发考勤重算 |
| BR-007 | 补卡申请审批通过但已封板超窗 | 仅允许通过调整单修正 |
| BR-008 | 结算单状态非 `APPROVED` | 禁止发起支付批次 |
| BR-009 | 重复 `X-Idempotency-Key` 请求 | 返回首次处理结果，不重复执行 |
| BR-010 | 发票申请驳回 | 不生成应收台账 |
| BR-011 | 核销累计金额 > 开票金额 | 拒绝核销并返回业务错误 |
| BR-012 | 消息消费失败 | 进入重试队列，超过阈值进入 DLQ |

## 7. 数据词典（核心）

| 术语 | 定义 | 英文字段建议 |
|---|---|---|
| 客户线索 | 尚未完全签约的潜在合作单位信息 | `lead` |
| 用工单位 | 与劳务公司合作的甲方单位 | `employer_unit` |
| 合同 | 约定服务、计费、结算与支付条款的法律载体 | `labor_contract` |
| 结算规则 | 计时/按天/计件/按月等计费模型配置 | `settlement_rule` |
| 考勤记录 | 员工每日出勤及工时记录 | `attendance_record` |
| 补卡申请 | 对异常考勤记录的修正申请 | `attendance_correction` |
| 结算单 | 一个结算周期内的汇总账单 | `settlement_order` |
| 支付批次 | 对结算单执行的打款任务集合 | `payment_batch` |
| 发票申请 | 对结算金额开票的申请单 | `invoice_application` |
| 应收台账 | 应收金额的跟踪账本 | `receivable_ledger` |
| 核销 | 回款对发票/应收进行抵扣确认 | `receivable_reconcile` |
| 数据权限 | 按组织/项目/客户限定查询范围 | `data_scope_policy` |
