# 测试策略（Stage 1）

> 依据 `docs/specification.md` 的 NFR Matrix 与当前技术栈制定。

## 一、测试金字塔

```text
        ┌────────┐
        │  E2E   │  10%  核心业务闭环（线索→合同→结算→发票）
        ├────────┤
        │ 集成测试 │  25%  API + DB + MQ + Redis 协同
        ├────────┤
        │ 单元测试 │  65%  业务规则、计算逻辑、权限校验
        └────────┘
```

## 二、覆盖率目标

| 层级 | 覆盖率目标 | 强制执行 |
|---|---|---|
| 后端单元测试 | 行覆盖率 >= 80% | ✅ CI 阻断 |
| 前端组件测试 | 行覆盖率 >= 60% | ✅ CI 阻断 |
| API 集成测试 | 核心接口 100% 覆盖 | ✅ CI 阻断 |
| E2E 测试 | 主链路场景 100% 覆盖 | ⚠️ 阶段门控 |

## 三、性能基线（来自 NFR）

| 指标 | 基线值 | 测试方法 |
|---|---|---|
| API P95 响应时间 | < 300ms | k6 压测 |
| API P99 响应时间 | < 800ms | k6 压测 |
| 页面首屏加载 | < 2.5s | Lighthouse |
| 可用率 | >= 99.9% | SLI 监控 |

## 四、测试框架选型

| 层级 | 技术栈 | 工具 |
|---|---|---|
| 后端单元 | Java/Spring Boot | JUnit 5 + Mockito |
| 后端集成 | Java/Spring Boot | Spring Boot Test + Testcontainers |
| 合约测试 | REST API | Spring Cloud Contract 或 Pact |
| 前端单元 | Vue 3 | Vitest + Vue Test Utils |
| 前端 E2E | Vue 3 | Playwright |
| 性能测试 | 全链路 | k6 |
| 安全扫描 | 代码/密钥/依赖 | gitleaks + OWASP Dependency-Check |

## 五、测试数据策略

1. 单元测试全部使用 Mock，不依赖外部服务。  
2. 集成测试优先使用 Testcontainers，保证与生产依赖一致。  
3. E2E 使用隔离测试库，执行前重置并加载种子数据。  
4. 金额与结算相关用例必须提供固定数据夹具，避免浮动误差。

## 六、关键业务回归清单

1. 认证与数据权限：越权拦截、菜单权限、接口权限。  
2. 合同状态机：创建、审批、锁定、终止、续签。  
3. 考勤补卡：提交、审批、封板内重算、封板外调整单。  
4. 结算计算：计时/按天/计件/按月四类模型。  
5. 支付幂等：重复请求不重复扣款。  
6. 发票与核销：分次核销且不超额。

## 七、命名与目录约定

| 场景 | 约定 |
|---|---|
| 后端单元测试 | `*Test.java` |
| 后端集成测试 | `*IntegrationTest.java` |
| 前端组件测试 | `*.spec.ts` |
| E2E 测试 | `*.e2e.spec.ts` |

## 八、CI 质量门

1. `frontend`: `npm ci && npm run build && npm run test`（后续补充 test 脚本）  
2. `backend`: `mvn -B test`  
3. `secrets-scan`: gitleaks  
4. 覆盖率门禁：低于阈值时阻断合并

## 九、风险与补救

| 风险 | 影响 | 对策 |
|---|---|---|
| Docker 环境不可用 | 集成测试无法执行 | 本地与 CI 均提供容器环境探活脚本 |
| JDK 版本不一致 | 测试结果偏差 | 统一使用 JDK 21 作为测试与构建标准 |
| 规则变更频繁 | 用例失效与回归缺口 | 建立合约测试与数据夹具版本化 |
