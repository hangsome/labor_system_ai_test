# Stage 2：阶段规划

## 一、目标

将系统开发工作按模块依赖和业务优先级，划分为 3-6 个可独立交付的开发阶段，每个阶段生成独立的 phase 文件夹。

## 二、AI 行为约定

1. 阶段划分必须遵循**依赖关系**：被依赖的模块优先开发。
2. 每个阶段的范围必须是**可独立验收**的：阶段结束后有可运行/可演示的交付物。
3. 使用 `sequential-thinking`（totalThoughts: 6-8）进行阶段划分推理。
4. 阶段总数控制在 **3-6 个**，避免过度拆分。
5. 阶段规划必须显式标注**演进任务**：Schema/API/架构文档刷新与本地基础设施演进不得缺失。
6. 若某 Phase 引入数据库结构变更，必须在该 Phase 规划中标注 migration 与 schema 文档同步。
7. 若某 Phase 引入新中间件，必须在该 Phase 规划中标注 `docker-compose.dev.yml` 演进任务。

## 三、阶段划分原则

### 依赖优先原则

```
Phase 1: 基础设施 + 公共组件（数据库、认证、基础 API）
Phase 2: 核心业务模块（最有业务价值的功能）
Phase 3: 辅助业务模块（次要功能）
Phase 4: 集成与优化（跨模块联调、性能优化）
Phase 5: 收尾（文档、部署、上线准备）
```

### 典型阶段划分模板

**中后台管理系统示例**：
| Phase | 名称 | 范围 | 前端 | 后端 |
|-------|------|------|------|------|
| 1 | 基础框架 | 项目框架 + 用户认证 + 权限管理 | 布局/登录页/菜单 | 认证API/用户CRUD |
| 2 | 核心业务A | 主要业务模块 | 业务页面/表单 | 业务API/数据处理 |
| 3 | 核心业务B | 次要业务模块 | 更多业务页面 | 更多API |
| 4 | 报表与统计 | 数据可视化 + 导出 | 图表/报表页 | 统计API/导出 |
| 5 | 优化与部署 | 性能优化 + 部署配置 | 优化/国际化 | 缓存/监控 |

**电商系统示例**：
| Phase | 名称 | 范围 |
|-------|------|------|
| 1 | 用户与认证 | 注册/登录/个人信息 |
| 2 | 商品管理 | 商品CRUD/分类/搜索 |
| 3 | 购物车与订单 | 购物车/下单/订单管理 |
| 4 | 支付与物流 | 支付集成/物流跟踪 |
| 5 | 运营后台 | 数据统计/营销工具/系统配置 |

## 四、执行步骤

### Step 1：分析模块依赖

从 `docs/architecture.md` 中提取模块列表和依赖关系，构建依赖图：

```
模块A → 模块B (A 依赖 B)
模块C → 模块B (C 依赖 B)
模块D → (无依赖)
```

### Step 2：划分阶段

使用 `sequential-thinking`（totalThoughts: 6-8）：

```
thought 1: 梳理所有模块和依赖关系
thought 2: 识别无依赖的基础模块（必须最先开发）
thought 3: 按业务优先级排列剩余模块
thought 4: 将模块分组为 3-6 个阶段
thought 5: 验证每个阶段的独立可交付性
thought 6: 确定每个阶段的测试策略
thought 7: 标注每个阶段的文档刷新与基础设施演进任务
```

### Step 3：生成 phase-plan.md

```markdown
# 阶段开发计划

## 总览
| Phase | 名称 | 模块 | 预估工作量 | 前置依赖 |
|-------|------|------|-----------|---------|
| 1 | ... | ... | ... | — |
| 2 | ... | ... | ... | Phase 1 |
| ... | ... | ... | ... | ... |

## Phase 1: <名称>

### 范围
- 前端：<包含的页面/组件>
- 后端：<包含的 API/服务>
- 数据库：<涉及的表>

### 交付目标
<阶段完成后可运行/验证的功能>

### 验收标准
1. ...
2. ...

### 演进与同步
- Schema 变更：<无 / migration + schema 文档更新>
- API/架构文档：<无 / api-contracts + architecture 增量更新>
- 本地依赖：<无 / docker-compose.dev.yml 变更>

## Phase 2: <名称>
...（同上结构）
```

### Step 4：生成 phase 目录结构

为每个阶段创建空的目录框架：

```
phases/
├── phase-01-<slug>/
│   ├── plan.md          # 阶段计划（placeholder）
│   ├── todolist.csv     # 任务清单（Stage 3 填充）
│   ├── process.md       # 进度跟踪（Stage 3 填充）
│   ├── feature-tasks/   # 特性任务目录
│   └── review/          # 审查报告目录
├── phase-02-<slug>/
│   └── ...
└── ...
```

**slug 命名规则**：
- 使用阶段核心主题的英文或拼音缩写
- 小写 + 连字符
- 示例：`phase-01-foundation`、`phase-02-core-business`

## 五、完成条件

- [ ] `docs/phase-plan.md` 已生成
- [ ] Phase 目录结构已创建
- [ ] 每个 Phase 有明确的范围和交付目标
- [ ] 阶段间依赖关系合理（无循环依赖）
- [ ] 每个 Phase 已标注文档刷新与基础设施演进策略（如适用）
- [ ] 用户确认阶段划分方案

## 六、输出到下一 Stage

- `docs/phase-plan.md`（阶段总览）
- `phases/phase-XX-xxx/` 目录结构（空框架）
- 当前要执行的阶段编号（默认从 Phase 1 开始）

## 七、动态 Plan 追加协议 (Dynamic Plan Append)

当用户在项目进行中传入新的 Plan 文件或提出新需求时，本 Stage 支持 **增量追加** 模式：

### 规则
1. **已完成的 Phase 不受影响**：绝不修改或重新划分已通过验收的 Phase。
2. **未开始的 Phase 可调整**：如果新需求属于尚未开始的现有 Phase，可将其合并到该 Phase 的范围中。
3. **所有 Phase 已完成时追加新 Phase**：创建 `phase-(N+1)-xxx/` 目录，更新 `docs/phase-plan.md`。
4. **更新 phase-plan.md**：所有变更必须同步反映到 `docs/phase-plan.md` 中。

### 执行流程
```
用户传入新 Plan → 解析新需求 → 
  if (属于未开始的现有 Phase):
    合并到该 Phase 范围
  else:
    创建新的 Phase(N+1)
→ 更新 phase-plan.md → 回到 Stage 3 继续
```
