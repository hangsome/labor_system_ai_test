# 劳务系统 0→1 实施计划（后台版｜Java + Vue + MySQL）

## 一、摘要
本计划用于在 `ai-dev-workflow` 7-Stage 框架下，从 0 到 1 落地劳务系统“后台 Web 版”，技术栈固定为 `Java 21 + Spring Boot 3.3`、`Vue 3 + Vite`、`MySQL 8`，并按“全量并行”原则推进业务主链、财务链、组织能力链。

当前 Calicat MCP 已连通，但对你提供的 `file_id=2009807574801920000` 读取图层时权限不足（`加入文件频道失败：无权限`）。因此计划采用“可执行主计划 + Calicat 权限门禁回填”的双态方案：先按现有原型推进，Calicat 解锁后执行字段和交互差异回填，不阻塞整体开工。

## 二、范围与非范围
### 2.1 本期范围（V1）
1. 管理后台登录与权限体系（RBAC + 数据权限）。
2. 客户与用工单位管理。
3. 合同管理与结算规则配置。
4. 员工/人才库/入离职与档案。
5. 排班、打卡、补卡审核、考勤台账。
6. 薪资结算、支付批次、调整单。
7. 发票申请、应收台账、核销。
8. 资金监控看板、财务报表。
9. 培训管理、绩效管理、积分管理。
10. 审计日志、系统配置、基础字典。

### 2.2 非范围（V1 不做）
1. 零工端与用工方端小程序 UI。
2. 多租户 SaaS 化（V1 明确单租户部署）。
3. 复杂 BPM 引擎（Flowable/Camunda）集成。

## 三、技术与架构定案（已锁定）
1. 后端：`Spring Boot 3.3` 模块化单体，按领域包拆分，统一 REST API。
2. 前端：`Vue3 + TypeScript + Vite + Pinia + Vue Router + Element Plus`。
3. 数据库：`MySQL 8`，迁移工具 `Flyway`。
4. 缓存与锁：`Redis`（幂等键、分布式锁、热点看板缓存、Token 黑名单）。
5. 消息：`RabbitMQ`（topic 交换机）承载异步通知和解耦任务。
6. 文件存储：开发 `MinIO`，生产 `OSS`（合同附件、发票影像、考勤凭证）。
7. 鉴权：`JWT + Refresh Token`，Spring Security。
8. 容量目标：DAU 3-8 万、在线并发 2k-5k；模块化单体可承载。触发拆分阈值：持续 2 周 `P99>800ms` 或核心队列积压 > 10 分钟或并发 > 1 万。

## 四、关键架构决策（阻断项已清零）
1. 租户策略：V1 单租户，不引入 `tenant_id`；通过组织/项目/客户数据权限隔离。
2. 审批流：采用轻量自研审批表（不引入流程引擎）。
3. 结算规则模型：`HOURLY | PIECE | MIXED`，规则版本“下个结算周期生效”，历史结算不可回写。
4. 封板策略：周结每周一 02:00、月结每月 2 日 02:00 自动触发；触发后 24 小时内允许补卡重算，超窗仅走“调整单”。
5. 状态机固定：
   - `contract`: DRAFT → ACTIVE → EXPIRING → ARCHIVED / TERMINATED
   - `attendance_correction`: SUBMITTED → REVIEWING → APPROVED / REJECTED → CLOSED
   - `settlement_order`: DRAFT → PENDING_APPROVAL → APPROVED → LOCKED → PAID / CANCELLED
   - `payment_batch`: DRAFT → APPROVED → PAID / FAILED
   - `invoice_application`: DRAFT → SUBMITTED → APPROVED → ISSUED → RECONCILED / REJECTED
6. 数据权限档位：`ALL / DEPT / PROJECT / CLIENT / SELF`，由 MyBatis 数据权限拦截器注入查询条件。

## 五、核心数据模型（V1）
核心表清单：
1. `user_account`, `role`, `permission`, `role_permission`, `user_role`
2. `org_department`, `org_position`, `data_scope_policy`
3. `customer`, `employer_unit`, `project`
4. `labor_contract`, `contract_attachment`, `settlement_rule`
5. `employee_profile`, `employee_bank_account`, `employee_insurance`, `employee_status_log`
6. `shift_schedule`, `attendance_record`, `attendance_correction`
7. `settlement_order`, `settlement_item`, `settlement_adjustment`
8. `payment_batch`, `payment_record`
9. `invoice_application`, `invoice_issue_record`, `receivable_ledger`, `receivable_reconcile`
10. `fund_ledger`, `cashflow_snapshot`
11. `training_plan`, `training_record`, `performance_plan`, `performance_record`, `points_rule`, `points_ledger`
12. `approval_flow`, `approval_node`, `approval_record`
13. `audit_log`, `sys_dict`, `sys_config`

关键约束：
1. `attendance_record` 唯一键：`(employee_id, work_date, shift_id)`。
2. `settlement_item` 必须归属 `settlement_order`，禁止孤立明细。
3. `payment_record` 必须引用已 `APPROVED/LOCKED` 的 `settlement_order`。
4. `receivable_reconcile` 允许对一个发票申请多次核销，累计核销额不可超开票额。
5. 所有金额字段统一 `DECIMAL(18,2)`，统一币种 `CNY`。

## 六、公共 API / 接口 / 类型约定（重要变更）
### 6.1 API 规范
1. 前缀统一：`/api/admin/v1`。
2. 写接口需支持 `X-Idempotency-Key`（结算、支付、发票提交、核销）。
3. 统一响应：
   - `ApiResponse<T> { code, message, data, traceId, timestamp }`
   - `PageResult<T> { records, pageNo, pageSize, total }`
4. 统一错误码分层：认证、授权、参数、状态机冲突、业务规则冲突、外部依赖失败。

### 6.2 关键接口组
1. `auth/*`：登录、刷新、登出、当前用户信息。
2. `rbac/*`：角色、权限、数据权限策略。
3. `customers/*`, `employer-units/*`, `projects/*`。
4. `contracts/*`, `settlement-rules/*`。
5. `employees/*`, `employee-status/*`, `employee-bank/*`。
6. `schedules/*`, `attendance/*`, `attendance-corrections/*`。
7. `settlements/*`, `settlement-adjustments/*`。
8. `payment-batches/*`, `payments/*`。
9. `invoices/*`, `receivables/*`, `reconciliations/*`。
10. `fund-ledgers/*`, `dashboards/*`。
11. `training/*`, `performance/*`, `points/*`。
12. `approvals/*`, `audit-logs/*`, `configs/*`, `dicts/*`。

### 6.3 消息契约
1. Exchange：`labor.topic`。
2. Routing keys：
   - `settlement.completed`
   - `payment.status.changed`
   - `attendance.abnormal`
   - `invoice.status.changed`
   - `audit.action`
3. 失败重试：`1m/5m/15m` 指数退避 + DLQ。

## 七、7-Stage 执行蓝图（与 ai-dev-workflow 对齐）
### Stage 0 项目初始化（1.5 周）
1. 产出 `docs/specification.md`，固化用户故事、业务规则、NFR、数据词典。
2. 建立脚手架：后端、前端、数据库、Redis、RabbitMQ、MinIO、Flyway。
3. 建立最小 CI：lint + unit + secrets-scan。
4. **Calicat 门禁任务**：
   - 任务 A：重新拉取 `get_meta_data/get_design_data/get_prd_list`。
   - 任务 B：产出“Calicat 差异表”（字段/状态/交互差异）。
   - 任务 C：差异映射到 `specification` 与 `api-contracts`。
5. Stage 0 DoD：能本地一键启动，权限门禁任务有结论（通过或记录阻塞和补救）。

### Stage 1 架构设计（1.5 周）
1. 完成 `architecture/database-schema/api-contracts/test-strategy`。
2. 固化状态机图、审批流图、消息拓扑图。
3. 固化 RBAC 权限矩阵和数据权限策略。
4. Claude 架构评审通过后进入 Stage 2。

### Stage 2 阶段规划（0.5 周）
1. 生成 6 个 Phase，按依赖分层并行推进三条工作流。
2. 为每个 Phase 创建 `plan/todolist/process` 模板。

### Stage 3/4/5 循环执行（每个 Phase 约 1.5-2 周）
1. Stage 3：拆原子任务，补齐验收标准与验证命令。
2. Stage 4：实现 + 测试 + 文档同步 + migration 同步。
3. Stage 5：代码审查 + 回归 + 人工验收门。

### Stage 6 部署与运维就绪（1 周）
1. 发布流程、回滚策略、监控告警、容量压测报告。
2. 输出单租户到 SaaS 化迁移预案（不实施，仅文档化）。

## 八、Phase 划分（全量并行执行）
1. Phase 1：平台底座。交付 RBAC、组织架构、审批框架、审计日志、字典配置。
2. Phase 2：主数据域。交付客户/用工单位/项目/合同/结算规则。
3. Phase 3：人力与考勤。交付员工全生命周期、排班、打卡、补卡审核。
4. Phase 4：结算与支付。交付结算单、明细、调整单、支付批次、幂等保障。
5. Phase 5：发票与资金。交付发票申请、开票记录、应收台账、核销、资金看板。
6. Phase 6：能力增强与收官。交付培训/绩效/积分、综合报表、上线验收与运维基线。

## 九、测试用例与验收场景
1. 认证：Token 刷新、过期、并发登录踢出策略。
2. RBAC：按钮权限、API 权限、数据权限越权拦截。
3. 合同：创建、变更、到期、归档、终止流程。
4. 结算规则：计时/计件/混合三模型计算正确性。
5. 考勤：重复打卡幂等、防重、跨天班次处理。
6. 补卡：提交、审批、驳回、重提、封板窗口内重算。
7. 结算：周结/月结自动触发与手动补触发一致性。
8. 调整单：封板后调整入账与审计可追溯。
9. 支付：重复请求不重复扣款，失败重试不重复入账。
10. 发票：申请、审批、开票、核销部分回款、全额回款。
11. 消息：消费者失败重试、死信回放、幂等消费。
12. 报表：合同、结算、应收、资金四类口径一致性。
13. 性能：结算批量任务、报表查询、并发审批压测。
14. 灾备：Redis/RabbitMQ 不可用时的降级与恢复。
15. Calicat 差异回归：回填后关键页面字段与流程一致性校验。

## 十、运维与监控
1. 指标：接口 P95/P99、慢 SQL、队列积压、任务失败率、审批耗时、结算耗时。
2. 告警：支付失败率阈值、结算任务超时、核销异常、权限越权审计。
3. 回滚：数据库 migration 可回滚脚本、应用灰度发布、支付流程补偿机制。

## 十一、显式假设与默认值
1. 已确认技术栈：`Java 后端 + Vue 前端 + MySQL 数据库`。
2. 已确认前端范围：仅劳务系统后台 Web。
3. 已确认规模目标：中高负载（DAU 3-8 万，并发 2k-5k）。
4. Calicat 当前在本账号下存在读取权限问题；本计划默认先按现有原型推进，Calicat 解锁后执行回填，不改变主架构。
5. 模块化单体在上述目标可承载；当并发或延迟超过阈值时，优先拆分 `结算域`、`发票资金域`、`考勤域`。
